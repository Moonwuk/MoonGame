# Журнал ревью и аудитов

> Постоянная папка, куда складываются находки роли **«поиск багов / стандартизация /
> упрощение»** и аудиты документации. Накопительный лог: новые проходы добавляются
> отдельными датированными файлами, старые не переписываются — статус правится на месте.
>
> Цель — чтобы команда могла **читать**, что нашли, в каком это состоянии и что
> предлагается, не перечитывая код заново. Каждая находка верифицирована против
> реального кода (правило проекта: «verify docs against reality before writing them»).

## Как читать

- Находки сгруппированы по проходам (один файл = один проход).
- У каждой находки есть **ID**, **серьёзность**, **статус**, точная **локация** (`file:line`),
  механизм («что и почему») и **предлагаемый фикс**.
- Ничего из кода по этим находкам ещё **не менялось** — это отчёты, а не правки
  (если не указано иное в колонке статуса).

### Легенда статусов

| Значок | Значение                                                        |
| ------ | --------------------------------------------------------------- |
| 🔴     | Open — подтверждено, не исправлено                              |
| 🟡     | In progress — взято в работу                                    |
| ✅     | Fixed — исправлено (указать коммит/PR)                          |
| ⚪     | Accepted / Won't-fix — осознанно оставлено (с обоснованием)     |

### Серьёзность

`HIGH` — заметный эффект / вводит в заблуждение · `MED` — реальный, но локальный ·
`LOW` — латентный / косметический.

## Как пополнять

1. Новый проход → новый файл `YYYY-MM-DD-<тема>.md`. Добавить строку в таблицу выше.
2. Каждая находка: уникальный ID (`<область>-<слаг>`), серьёзность, статус, `file:line`,
   механизм, фикс. Перед записью — **сверить с актуальным кодом**, не по памяти.
3. Когда находка исправлена — поменять её статус на ✅ и приписать коммит/PR (файл-источник
   не переписывать, чтобы история находок сохранялась).

## Метод

Находки собраны многоагентным проходом по подсистемам с **состязательной верификацией**
каждой (независимый агент пытался опровергнуть находку, читая реальный код; в отчёт попали
только подтверждённые). Базовая линия на момент аудита: `pnpm run check` зелёный —
**282 теста / 37 файлов**, lint + typecheck чисты.
